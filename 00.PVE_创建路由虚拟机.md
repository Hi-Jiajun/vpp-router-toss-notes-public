# PVE 创建路由虚拟机

这一步记录当前这套路由系统在 `PVE` 中的虚拟机创建方式，作为后续 `Ubuntu + VPP` 配置的基础。

本文基于当前实际使用的路由虚拟机：

- `VM ID`: `103`
- 虚拟机名称：`UbuntuForRouter`
- 宿主平台：`Proxmox VE`

## 0. 背景

当前这套方案不是把 `Ubuntu` 直接安装到物理机，而是先在 `PVE` 中创建一台专门承担路由功能的虚拟机，然后在虚拟机内部运行：

- `VPP`
- `dnsmasq`
- 路由相关的 Linux 控制面服务

这样做的目的主要有两点：

1. 宿主机和路由系统解耦，后续升级和回滚更方便。
2. 可以结合 `PCIe Passthrough`、`CPU pinning`、`hugepages` 等能力，把虚拟化开销尽量压低。

## 1. 前期准备

在创建虚拟机前，默认已经满足这些条件：

- `PVE` 已正常安装并可进入 Web 管理界面
- 宿主机已经开启 `IOMMU`
- 计划直通给路由虚拟机的网卡功能已经可以被 `VFIO` 接管
- 已准备好 `Ubuntu Server` 安装镜像

当前这台路由虚拟机在 `PVE` 宿主机侧直通了两个 PCIe 设备功能：

- `0000:01:00.2`
- `0000:01:01.2`

注意：

- 这是 `PVE` 宿主机侧看到的地址
- 进入 `Ubuntu` 或 `VPP` 后，设备的 `PCI BDF` 可能会重新编号
- 后续 `03` 和 `04` 章节中的 `PCI` 地址，应以客户机内部实际识别结果为准

## 2. 创建 Linux 虚拟机

### 2.1 常规

创建虚拟机时，当前建议至少确认这些参数：

- `VM ID`: `103`
- 名称：`UbuntuForRouter`
- 开机自启动：开启
- 启动顺序：`order=1`
- 启动延时：`up=10`
- 标签：`network;router`

### 2.2 操作系统

当前虚拟机用于安装 `Ubuntu Server`，类型可按 Linux 虚拟机处理。

本文最终实际配置使用：

- `bios: ovmf`
- `ostype: l26`

如果后续需要：

- 使用 `UEFI`
- 配合现代 `PCIe` 设备直通

那么 `OVMF + q35` 会比传统的 `SeaBIOS + i440fx` 更顺手。

### 2.3 系统

当前实际系统相关参数如下：

- `machine: q35`
- `agent: 1`
- `tablet: 0`
- `rng0: source=/dev/urandom`
- `efidisk0` 已创建

其中：

- `q35` 用于更好支持 `PCIe` 设备直通
- `QEMU Agent` 开启后，宿主机获取来宾状态会更方便
- `tablet: 0` 可以减少不必要的虚拟输入设备

### 2.4 磁盘

当前系统盘配置如下：

- `scsi0: local:103/vm-103-disk-1.qcow2`
- 容量：`32G`
- `discard=on`
- `iothread=1`
- `ssd=1`
- `scsihw: virtio-scsi-single`

引导顺序当前为：

```text
boot: order=scsi0
```

也就是只从系统盘引导，不保留网络启动项。

### 2.5 CPU

当前 CPU 相关配置如下：

- `cpu: host`
- `sockets: 1`
- `cores: 5`
- `affinity: 2,4,6,8,10`
- `numa: 1`

这组参数的核心思路是：

- 让虚拟机直接吃到宿主机 CPU 特性
- 给路由虚拟机固定一组物理线程
- 避免与宿主机其他负载频繁争抢 CPU

### 2.6 内存

当前内存相关配置如下：

- `memory: 6144`
- `balloon: 0`
- `hugepages: 1024`

其中：

- `balloon: 0` 表示关闭动态回收内存
- `hugepages: 1024` 表示使用 `1G hugepages`

这类配置更适合长期常驻、对性能波动敏感的路由虚拟机。

### 2.7 网络

当前这台虚拟机没有使用普通 `VirtIO` 虚拟网卡，而是直接做 `PCIe Passthrough`：

- `hostpci0: 0000:01:00.2,pcie=1,rombar=0`
- `hostpci1: 0000:01:01.2,pcie=1,rombar=0`

当前思路是：

- `WAN` 和 `LAN` 都交给直通网卡处理
- 尽量避免关键数据面流量经过宿主机桥接转发
- 让 `VPP` 在虚拟机内部直接接管网卡

### 2.8 确认

创建完成后，至少应确认：

- 虚拟机能正常开机
- 系统盘是第一启动项
- `QEMU Agent` 已开启
- 两个 `hostpci` 设备都已挂载到虚拟机
- 自启动、启动顺序、标签都符合预期

## 3. 当前实际配置文件

当前 `PVE` 宿主机上的配置文件内容如下：

文件：

- `/etc/pve/qemu-server/103.conf`

内容：

```conf
affinity: 2,4,6,8,10
agent: 1
balloon: 0
bios: ovmf
boot: order=scsi0
cores: 5
cpu: host
efidisk0: local:103/vm-103-disk-0.qcow2,efitype=4m,size=528K
hostpci0: 0000:01:00.2,pcie=1,rombar=0
hostpci1: 0000:01:01.2,pcie=1,rombar=0
hugepages: 1024
machine: q35
memory: 6144
meta: creation-qemu=10.1.2,ctime=1772614052
name: UbuntuForRouter
numa: 1
onboot: 1
ostype: l26
rng0: source=/dev/urandom
scsi0: local:103/vm-103-disk-1.qcow2,discard=on,iothread=1,size=32G,ssd=1
scsihw: virtio-scsi-single
smbios1: uuid=11111111-2222-3333-4444-555555555555
sockets: 1
startup: order=1,up=10
tablet: 0
tags: network;router
vmgenid: aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee
```

## 4. 关键参数说明

- `cpu: host`
  - 让客户机尽量直接使用宿主机 CPU 能力，对 `VPP` 这类偏性能敏感的场景更友好
- `affinity`
  - 把这台路由虚拟机固定到一组 CPU 线程，减少调度抖动
- `hugepages: 1024`
  - 与后续 `VPP` 使用大页内存的思路一致
- `balloon: 0`
  - 路由虚拟机不适合被宿主机动态挤压内存
- `q35 + ovmf`
  - 对现代 `PCIe` 设备直通更友好
- `hostpci0 / hostpci1`
  - 表示当前是“网卡直通”而不是“桥接虚拟网卡”
- `boot: order=scsi0`
  - 明确只从系统盘启动，避免把网络启动项带进来
- `tablet: 0`
  - 关闭虚拟平板设备，减少无关虚拟硬件

## 5. 创建后核对

虚拟机安装完成并首次启动后，建议先核对这些点：

### 5.1 PVE 侧

- 虚拟机可正常启动和关机
- `QEMU Agent` 状态正常
- 两个 `hostpci` 设备仍然处于已分配状态
- 自启动和启动顺序符合预期

### 5.2 Ubuntu 侧

```bash
lspci -nn
ip -br link
systemctl status qemu-guest-agent
```

重点确认：

- 两块直通网卡在客户机内都能被识别
- 客户机内的 `PCI` 地址可能与宿主机不同
- 后续 `VPP` 配置应使用客户机内部实际识别到的地址

### 5.3 VPP 侧

等 `VPP` 配置完成后，再核对：

```bash
vppctl show hardware-interfaces
vppctl show interface
```

如果 `VPP` 中识别到的 `PCI` 地址与 `PVE hostpci` 行不同，这通常是正常现象，按客户机内部结果继续配置即可。
