# 拓扑与地址规划

## 物理/虚拟接口

- `enp1s0`
  - VPP 识别为 `wan0`
  - PCI: `0000:01:00.0`
- `enp2s0`
  - VPP 识别为 `lan0phy`
  - PCI: `0000:02:00.0`
- `lan0`
  - 由 `LCP` 创建出来的 Linux host 接口
- `hosttap0 / vpphost0`
  - Linux 通过 VPP 出网的专用内部通道

## 地址规划

### LAN IPv4

- `192.168.1.1/24`

### LAN GUA

- 动态，来自 `DHCPv6 PD`
- 当前示例：`2001:db8:1000:1::1/64`

### LAN ULA

- `fd00:1234:5678:1::1/64`

### Linux 经 VPP 出网内部通道

- VPP: `10.255.255.1/30`
- Linux: `10.255.255.2/30`

## 关键下一跳

这些地址背后分别对应的内网辅助项目，见：

- [11.内网辅助节点说明](./11.内网辅助节点说明.md)

### fakeip

- `7.0.0.0/8 -> 192.168.1.4`

### Telegram IPv4

- 官方 IPv4 CIDR -> `192.168.1.4`

### Telegram IPv6

- 官方 IPv6 CIDR -> `fd00:1234:5678:1::254`
- 与 `192.168.1.4` 为同一台设备的 `IPv6` 地址

## 设计原则

1. `VPP` 负责主路由数据面，不让 Linux 内核承担真正的转发角色。
2. Linux 只负责控制面，并通过 `vpphost0` 间接借 VPP 出网。
3. `192.168.1.4` 只承担 fakeip / Telegram 相关特殊下一跳，不承担 LAN 的默认 IPv6 router 角色。
4. 管理口不能再和业务 LAN 处于同一个二层广播域。
