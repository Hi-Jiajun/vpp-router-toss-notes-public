# Ubuntu 设置系统网络

本章记录当前这套路由虚拟机在安装完系统参数后，Linux 侧网络的职责边界和基本设置思路。

参考结构来源：

- [04.Ubuntu_设置系统网络.md](https://gitee.com/callmer/linux_router_toss_notes/blob/main/04.Ubuntu_%E8%AE%BE%E7%BD%AE%E7%B3%BB%E7%BB%9F%E7%BD%91%E7%BB%9C.md)

但当前环境和参考文档的差异非常大：

- 当前不是传统的 `Linux bridge + PPPoE + nftables` 路由方案
- 当前数据面由 `VPP` 接管
- Linux 只承担控制面、辅助服务和主机自身维护出网

因此，这一章不会照着参考文档去展开：

- `bridge`
- `PPPoE`
- `nftables`
- `SmartDNS`

这些不属于当前方案的核心 Linux 网络职责。

本文只记录在当前架构里真正需要明确的 Linux 侧网络边界。

## 1. 设置网络服务

当前环境仍然保留：

- `systemd-networkd`
- `netplan`

但它们在这套方案里的职责比较克制：

- `systemd-networkd` 负责基础网络服务框架
- `netplan` 主要保留为 Linux 侧模板和兜底配置
- 不把核心数据面逻辑放在 Linux 网络服务里实现

可以先核对基础网络服务状态：

```bash
# 确认 networkd 处于启用状态
systemctl status systemd-networkd.service

# 如果计划让 dnsmasq 直接占用 53 端口，再确认 resolved 当前状态
systemctl status systemd-resolved.service
```

如果后续要让 `dnsmasq` 直接监听 `53` 端口，通常需要停用 `systemd-resolved`：

```bash
# 立即停止 resolved，释放 53 端口
sudo systemctl stop systemd-resolved.service
# 禁止开机自动启动
sudo systemctl disable systemd-resolved.service
# 进一步 mask，避免其他组件重新拉起
sudo systemctl mask systemd-resolved.service
```

## 2. 设置物理接口边界

当前虚拟机内部能看到两块直通网卡：

- `enp1s0`
- `enp2s0`

但在当前方案里，它们的角色不是“由 Linux 自己配置成普通网口”，而是：

- 作为客户机内部可见的物理直通设备
- 最终交由 `VPP` 识别和接管

因此，这里最重要的原则不是“如何给它们配 IP”，而是：

1. 不让 Linux 在其上做 DHCP 自动配置。
2. 不让 Linux 在其上接受 RA 或自动生成 IPv6 地址。
3. 不在 Linux 里把它们组桥、做 PPPoE、做默认路由。

这些约束已经在上一章的系统参数和 `netplan` 模板中体现出来。

## 3. 设置 Linux 控制面接口

当前这套方案里，Linux 真正会用到的关键接口不是直通物理口本身，而是后续由 `VPP` 派生或协同产生的接口。

### 3.1 `lan0`

`lan0` 是 Linux 侧能看到的 `host` 接口，由后续 `VPP LCP` 创建。

它的主要用途是：

- 承接 Linux 侧的 `dnsmasq`
- 让 Linux 能感知和服务 LAN

但注意：

- `lan0` 不是 Linux 自己创建出来的主桥接口
- 其基础地址和真实数据面语义仍由 `VPP` 章节决定

### 3.2 `vpphost0`

`vpphost0` 是 Linux 通过 `VPP` 出网的内部控制面接口。

它的主要用途是：

- `apt update`
- `curl`
- `git clone`
- 其他主机自身维护动作

也就是说，Linux 主机自己的默认路由最终不是直接压在某块物理网卡上，而是走这条内部通道。

## 4. 当前网络职责划分

为了避免后续配置时混淆，这里把当前方案里各层职责再明确一次：

### 4.1 Linux 负责

- 提供控制面环境
- 运行 `dnsmasq`
- 运行本机 `mini-ppdns`
- 通过 `vpphost0` 获取主机自身的联网能力
- 保持系统网络行为可控、安静

### 4.2 VPP 负责

- 识别并接管直通网卡
- 配置 `WAN / LAN`
- `PPPoE`
- `NAT44 EI`
- `IPv6 PD`
- 创建 `LCP` 和 `tap` 相关接口

## 7. 基础核对命令

如果只想确认 Linux 侧网络状态有没有跑偏，可以先看这些：

```bash
# 检查 resolved 是否已经停用
systemctl status systemd-resolved.service
# 查看 networkd 识别到的接口
networkctl
# 查看接口列表
ip -br link
# 查看接口地址
ip -br a
# 查看 IPv4 路由
ip route
# 查看 IPv6 路由
ip -6 route
```

等后续 `VPP` 配置完成后，再重点核对：

```bash
ip -br a show lan0
ip -br a show vpphost0
```
