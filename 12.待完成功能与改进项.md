# 待完成功能与改进项

本章用于记录这套路由方案当前还没做、但后续值得补齐的能力。

这些内容不一定都要立刻实现，但适合作为后续持续演进的 backlog。

## 🌐 网络功能

### 1.1 UPnP / NAT-PMP

当前这套方案里还没有补齐自动端口映射能力。

后续可考虑：

- 为内网客户端提供 `UPnP` / `NAT-PMP`
- 限制只允许可信网段或可信设备发起映射
- 对映射端口范围、协议类型、生命周期做约束

需要重点评估：

- 最终由 `VPP` 侧还是 Linux 控制面侧承接
- 如何避免和现有 `NAT44 EI` 行为冲突
- 如何避免把高风险映射直接暴露到公网

### 1.2 ACL 规则

当前主配置里已经启用了 `acl_plugin.so`，但还没有形成成体系的访问控制规则。

后续可考虑：

- 对不同网段或不同主机做访问控制
- 限制特定设备访问指定目标网段或端口
- 对管理面、旁路节点、基础设施服务做隔离

适合优先梳理的方向：

- 内网客户端到路由器管理面的访问
- 客户端到 `192.168.1.2 / 192.168.1.4 / 192.168.1.250` 这类辅助节点的访问范围
- IPv4 / IPv6 双栈下一致的访问控制策略

### 1.3 更细粒度的静态分流

当前已经做了：

- fakeip 静态路由
- Telegram IPv4 / IPv6 静态路由

后续还可以考虑：

- 把旁路规则收敛成更清晰的分类
- 区分“固定下一跳”“策略路由”“域名驱动分流”
- 为不同旁路节点预留独立的规则文件和说明

## 🛠️ 管理与可视化

### 2.1 Web 界面

当前这套方案基本完全依赖：

- `ssh`
- 配置文件
- `vppctl`
- `systemctl`

后续可考虑补一个轻量级 Web 界面，用于：

- 查看接口状态
- 查看 PPPoE / NAT / DHCP / DNS 状态
- 查看当前静态路由与旁路节点状态
- 做基础的开关、重载和只读展示

这里建议优先做“只读面板”，再考虑“可写配置”。

原因是：

- 只读面板更容易落地
- 风险更小
- 能先解决“状态分散、排障要多窗口切换”的问题

### 2.2 配置生成与集中管理

当前配置主要以内嵌文档和手工维护为主。

后续可考虑：

- 把关键配置抽成模板
- 通过脚本生成 `vpp` / `dnsmasq` / `sysctl` 配置
- 引入统一的变量文件，减少重复修改

尤其适合模板化的内容包括：

- 地址规划
- 旁路下一跳
- Telegram 路由文件
- `dnsmasq` 静态绑定

## 🛡️ 可靠性与安全性

### 3.1 服务健康检查

当前文档已经有排障命令，但还没有形成自动健康检查。

后续可考虑：

- 定时检查 PPPoE 状态
- 定时检查默认路由、NAT、DNS 服务状态
- 检查静态旁路节点是否可达
- 异常时记录快照，必要时再告警

### 3.2 更安全的自动更新

当前 Telegram 路由更新里，IPv4 脚本仍会自动 `restart vpp`。

后续值得改进：

- 更新前先生成新文件并校验
- 尽量改为无中断重载
- 把“下载失败 / 结果为空 / 规则异常”的场景挡在重载前

### 3.3 管理面收敛

后续可进一步明确：

- 哪些接口允许管理访问
- 哪些节点允许 `ssh`
- 是否需要单独管理 VLAN / 管理网段

如果后面补 ACL，这部分应该优先落地。

## 📡 DNS 体系演进

后续可考虑：

这部分最好和 [11.内网辅助节点说明](./11.内网辅助节点说明.md) 一起维护。
