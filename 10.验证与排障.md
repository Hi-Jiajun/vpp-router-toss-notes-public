# 验证与排障

## 1. 基础验证

### 1.1 VPP

```bash
vppctl show version
vppctl show pppoe client detail
vppctl show interface address
vppctl show nat44 ei interfaces
vppctl show lcp
```

### 1.2 Linux

```bash
ip -br a
ip route
ip -6 route
ping -c 3 223.5.5.5
getent hosts www.baidu.com
```

### 1.3 dnsmasq

```bash
dnsmasq --test
systemctl status dnsmasq
tail -n 50 /var/log/dnsmasq.log
```

## 2. Telegram 路由验证

```bash
vppctl show ip fib | grep '192.168.1.4'
vppctl show ip6 fib | grep 'fd00:1234:5678:1::254'
```

## 3. 故障前取证清单

当再次出现“客户端断流、怀疑需要重启 VPP”时，先不要重启，先跑：

```bash
date

echo '===== PPPoE ====='
vppctl show pppoe client detail

echo '===== IFADDR ====='
vppctl show interface address

echo '===== VPP PING4 ====='
vppctl ping 223.5.5.5

echo '===== VPP PING6 ====='
vppctl ping ipv6 2400:3200::1 source lan0phy repeat 3

echo '===== NAT SESSIONS ====='
vppctl show nat44 ei sessions

echo '===== INTERFACES ====='
vppctl show interface

echo '===== ERRORS ====='
vppctl show errors | sed -n '1,160p'

echo '===== VPP LOG TAIL ====='
tail -n 80 /var/log/vpp/vpp.log

echo '===== DNSMASQ LOG TAIL ====='
tail -n 80 /var/log/dnsmasq.log

echo '===== NEIGH6 ROUTER ====='
ip -6 neigh show dev lan0 | grep router || true
```

## 4. 历史踩坑

### 4.1 管理口和业务 LAN 混二层

这个问题曾导致：

- 客户端把 `192.168.1.1` ARP 到错误的 MAC
- 随机断流
- `i/o timeout`
- 重启后暂时恢复

原则：

- 管理口绝不能和业务 LAN 处于同一个二层广播域

### 4.2 `gateway-helper` 开启 `ip6=auto`

这个问题曾导致：

- `192.168.1.4` 被识别成第二个 IPv6 router
- 客户端默认 IPv6 路由被干扰

原则：

- `192.168.1.4` 不应承担整段 LAN 的 IPv6 router 角色

### 4.3 `/etc/dnsmasq.d/` 内放 `.bak` 文件

因为 `dnsmasq` 会扫描整个目录，备份文件可能会被一起加载，触发：

- `illegal repeated keyword`
- `only one tag allowed`

原则：

- 备份文件不要留在 `/etc/dnsmasq.d/`
- 备份可放到 `/root/dnsmasq-backup/`
